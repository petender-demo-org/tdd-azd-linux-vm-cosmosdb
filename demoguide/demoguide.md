## 3. SSH Into VM And Deploy The .NET App Manually

### Duration

- 8 minutes

### What To Show

- Manual app publish and copy to VM
- Systemd service setup on Ubuntu

### Commands (Local Machine)

```powershell
Set-Location c:\azd-demo-builder-v1\generated-scenarios\linux-vm-cosmosdb

# Build and publish the app locally
dotnet publish .\src\LinuxVmCosmosdb.Web\LinuxVmCosmosdb.Web.csproj -c Release -o .\publish

# Copy published files to VM
scp -r .\publish\* azureuser@$vmPublicIp:/home/azureuser/logistics-app/
```

### Commands (On The VM)

```bash
ssh azureuser@20.22.170.24

sudo mkdir -p /opt/logistics-app
sudo cp -r /home/azureuser/logistics-app/* /opt/logistics-app/
sudo chown -R azureuser:azureuser /opt/logistics-app

cat <<'EOF' | sudo tee /etc/systemd/system/logistics-web.service
[Unit]
Description=Linux VM Cosmos DB Logistics Demo Web App
After=network.target

[Service]
WorkingDirectory=/opt/logistics-app
ExecStart=/usr/bin/dotnet /opt/logistics-app/LinuxVmCosmosdb.Web.dll
Restart=always
RestartSec=5
User=azureuser
Environment=ASPNETCORE_ENVIRONMENT=Production
Environment=COSMOS_ENDPOINT=https://cosmos-linuxvm-linux-vm-cosmosdb-lbajpj.documents.azure.com:443/

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable logistics-web
sudo systemctl restart logistics-web
sudo systemctl status logistics-web --no-pager
```

### Validation

```powershell
Invoke-WebRequest -Uri "$appUrl/healthz" -UseBasicParsing | Select-Object StatusCode
Invoke-WebRequest -Uri $appUrl -UseBasicParsing | Select-Object StatusCode
```

### What To Say

- Deployment to VM is explicit and transparent: publish, copy, and run as a service.
- The app only needs Cosmos endpoint; authentication is performed by `DefaultAzureCredential`.

### If It Fails

```bash

sudo systemctl restart logistics-web
```

## 5. Show Managed Identity Auth (No Connection Strings)

### Duration

- 4 minutes

### What To Show

- RBAC assignment between VM managed identity and Cosmos DB
- Application code path that uses `DefaultAzureCredential`

### Commands

```powershell
# Show VM managed identity principal ID
az vm show -g $resourceGroup -n $vmName --query "identity.principalId" -o tsv

# Show Cosmos DB SQL role assignments
az cosmosdb sql role assignment list -g $resourceGroup -a $cosmosAccount --output table
```

### Code Proof Points

- App startup creates `CosmosClient(cosmosEndpoint, new DefaultAzureCredential())`.
- No key-based connection string is used by the application.

### What To Say

- Identity-based auth removes secret rotation overhead and reduces credential leakage risk.
- Access is enforced by Cosmos DB data-plane RBAC, scoped to the account.

## 6. Monitoring Via Log Analytics

### Duration

- 4 minutes

### What To Show

- Workspace ingestion and queries for VM and Cosmos diagnostics
- Optional health check telemetry during app requests

### Commands

```powershell
$workspaceId = az monitor log-analytics workspace show `
  -g $resourceGroup -n $logWorkspace --query customerId -o tsv

# Last 30 minutes Cosmos requests
az monitor log-analytics query --workspace $workspaceId `
  --analytics-query "AzureDiagnostics | where TimeGenerated > ago(30m) " `
  "| where ResourceProvider == 'MICROSOFT.DOCUMENTDB' | summarize Count=count() by OperationName" `
  --output table

# Last 30 minutes VM heartbeat
az monitor log-analytics query --workspace $workspaceId `
  --analytics-query "Heartbeat | where TimeGenerated > ago(30m) | summarize LastSeen=max(TimeGenerated) by Computer" `
  --output table
```

### What To Say

- Log Analytics centralizes signals from both compute and data tiers.
- This allows fast operational triage during live demos and real environments.

### If It Fails

- Confirm diagnostic settings are enabled for Cosmos DB and VM resources.
- Re-run query with `ago(2h)` if data ingestion is delayed.

## 9. Contingency Playbook

- Cannot SSH to VM
  - Quick diagnosis: `az network nsg rule list -g $resourceGroup --nsg-name $nsgName -o table`
  - Recovery: Add scoped SSH allow rule; validate with `Test-NetConnection $vmPublicIp -Port 22`
- App does not start
  - Quick diagnosis: `sudo systemctl status logistics-web --no-pager`
  - Recovery: Check journal, confirm .NET runtime and `COSMOS_ENDPOINT`, then restart service
- Cosmos auth errors
  - Quick diagnosis: `az cosmosdb sql role assignment list -g $resourceGroup -a $cosmosAccount -o table`
  - Recovery: Recreate role assignment for VM principal ID and restart app
- No monitoring data
  - Quick diagnosis: Query returns empty
  - Recovery: Wait 5-10 minutes, then verify diagnostic settings and workspace linkage
